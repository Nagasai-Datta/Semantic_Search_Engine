# Design Decisions & Justification

Every significant design decision, the evidence behind it, and alternatives
considered. All referenced plots are generated by `python analysis/justification.py`
and saved to `analysis/plots/`.

---

## 1. Data Cleaning

### What was removed and why

The 20 Newsgroups dataset contains raw internet posts from the 1990s. Each post
includes metadata headers (`From:`, `Subject:`, `Organization:`, `Lines:`) that
are not part of the document's content. Retaining these causes the embedding
model to associate email domains and usernames with topic categories rather than
learning from actual language.

Posts also contain quoted reply blocks (lines prefixed with `>`), which are text
authored by other users and do not represent the document's own content.

**Cleaning steps applied:**

| Artifact             | Removal method                                  | Reason                   |
| -------------------- | ----------------------------------------------- | ------------------------ |
| Metadata headers     | `sklearn remove=('headers','footers','quotes')` | No semantic value        |
| Quoted reply lines   | Regex `^>.*$` per line                          | Another author's words   |
| Email addresses      | Regex `\S+@\S+`                                 | Noise                    |
| URLs                 | Regex `http\S+`                                 | Noise                    |
| Non-ASCII characters | Regex `[^\x00-\x7F]`                            | 1990s encoding artifacts |

### Minimum word count threshold

Documents with fewer than 20 words after cleaning are discarded. Inspection of
dropped documents confirmed they are almost entirely empty posts, single-line
replies, or header-only artifacts. Such documents cannot produce meaningful
384-dimensional embeddings.

## 2. Embedding Model

### Choice: `all-MiniLM-L6-v2` (sentence-transformers)

**Alternatives considered:**

| Model               | Why rejected                                                      |
| ------------------- | ----------------------------------------------------------------- |
| Raw BERT            | Per-token vectors — no per-document output without manual pooling |
| `all-mpnet-base-v2` | 768-dim, 5x slower, marginal quality gain                         |
| TF-IDF              | Keyword frequency — cannot recognise paraphrases                  |

**Why `all-MiniLM-L6-v2`:**

- Produces one 384-dim vector per document directly
- Fine-tuned on 1B+ sentence pairs for semantic similarity
- 5x faster than larger alternatives with ~95% quality
- Industry standard for retrieval tasks of this scale

**Evidence:** `analysis/plots/embedding_similarity_demo.png`
Cosine similarity heatmap across hand-picked sentence pairs confirms that
space-related sentences are highly similar to each other and dissimilar to
unrelated sentences.

### Vector normalisation

All embeddings are produced with `normalize_embeddings=True` (unit vectors),
so cosine similarity reduces to a dot product — faster to compute at scale.

## 3. Vector Database

### Choice: ChromaDB

| Option              | Why not chosen                              |
| ------------------- | ------------------------------------------- |
| FAISS               | In-memory by default, no metadata filtering |
| Pinecone / Weaviate | Require external hosted infrastructure      |
| numpy brute force   | O(n) scan — does not scale                  |

**Why ChromaDB:**

- Pure Python library, zero infrastructure
- Persists to disk, survives restarts
- Stores metadata (category labels) alongside vectors
- Configured with `hnsw:space: cosine` — approximate nearest-neighbour, sub-linear query time

## 4. Clustering Algorithm

### Choice: PCA → Gaussian Mixture Model (GMM)

**Why soft/fuzzy clustering:**

Hard clustering (KMeans) assigns each document to exactly one cluster.
A post about gun legislation is semantically related to both politics and
firearms. Forcing a single label discards meaningful signal.

GMM produces a probability distribution per document:

```
"Senate votes on NASA funding"
  KMeans → politics  (forced, incorrect)
  GMM    → {politics: 0.55, space: 0.38, other: 0.07}  (correct)
```

**Why GMM over Fuzzy C-Means (skfuzzy):**

Both algorithms are mathematically equivalent in concept. skfuzzy's cmeans
was tested first but degenerated to uniform membership distributions
(FPC = exactly 1/k) regardless of the number of dimensions tried. This is a
known numerical stability issue in the skfuzzy implementation — the error
threshold is met on the first iteration before the algorithm has learned
anything. sklearn's GaussianMixture uses Expectation-Maximisation and
converges reliably.

**Why PCA before clustering:**

In 384 dimensions, pairwise Euclidean distances between all document pairs
converge to nearly the same value (curse of dimensionality). No clustering
algorithm can distinguish boundaries in this regime.

PCA reduces to 50 components, retaining dominant variance while restoring
meaningful distance geometry. The fitted PCA model is saved to
`data/pca_model.joblib` so that incoming query embeddings at API request
time can be projected into the same space for accurate cluster routing.

Full 384-dim vectors continue to be used by ChromaDB for semantic search
and by the cache for cosine similarity comparison. PCA is applied to
cluster assignment only.

### Fuzziness

GMM's `covariance_type='diag'` gives each component its own diagonal
covariance matrix. This is appropriate for independently varying embedding
dimensions and is faster than full covariance. `n_init=3` runs three
random initialisations and keeps the best result, reducing sensitivity
to the random seed.

## 5. Number of Clusters

### How the value was selected

`analysis/justification.py` fits GMM for `k ∈ {5, 10, 15, 20, 25, 30}`.

**BIC (Bayesian Information Criterion):** lower is better.
Penalises model complexity — the elbow in the BIC curve identifies the
most efficient k (best fit without overfitting).

**AIC (Akaike Information Criterion):** lower is better.
Less complexity penalty than BIC — used as a secondary reference.

**Silhouette Score:** higher is better.
Measures cluster separation on a 3,000-document subsample.

**Evidence:** `analysis/plots/elbow_curves.png`
BIC and Silhouette curves plotted against k. The recommended value is
marked at the elbow / peak.

### Semantic validation

**Evidence:** `analysis/plots/cluster_heatmap.png`
Heatmap of average GMM membership per original category per cluster.
Clusters lighting up for semantically related categories confirm
meaningful topic grouping.

**Evidence:** `analysis/plots/pca_clusters.png`
2D PCA projection coloured by dominant cluster.
Distinct colour regions confirm clusters occupy separate areas of the space.

**Evidence:** `analysis/plots/cluster_samples.txt`
Top-3 most representative documents per cluster — readable topic coherence.

## 6. Boundary Cases

**Evidence:** `analysis/plots/boundary_cases.txt`

Highest-entropy documents sit between cluster boundaries. Example:

```
category: talk.politics.guns   entropy: 2.41
  cluster_N (politics): 0.44
  cluster_M (firearms): 0.39
```

A post about gun legislation genuinely spans both topics. The soft model
captures this correctly; hard clustering would force an incorrect single label.

## 7. Semantic Cache

### Why semantic rather than exact-match

A key-value cache on query strings fails when the same question is phrased
differently. The semantic cache stores query embeddings — two queries are
equivalent if their cosine similarity exceeds the threshold.

### Two embedding spaces

| Space          | Dimensions | Used for                                          |
| -------------- | ---------- | ------------------------------------------------- |
| Full embedding | 384        | Cosine similarity comparison inside cache buckets |
| PCA-reduced    | 50         | Cluster routing (matching GMM center space)       |

The full embedding captures fine-grained semantic similarity.
The PCA-reduced embedding enables accurate cluster assignment by matching
the space the GMM was trained in.

### Cluster-aware routing

Without clustering: O(n) lookup — compare against every cached entry.
With k=15 clusters: O(n/k) — compare only within one bucket (~15x faster).

### Similarity threshold

**Evidence:** `analysis/plots/threshold_experiment.png`

| Threshold | Behaviour                                             |
| --------- | ----------------------------------------------------- |
| 0.60      | High hit rate — many false matches                    |
| 0.75      | Paraphrases hit; distinct queries miss — chosen value |
| 0.95      | Almost no hits — paraphrases not recognised           |

The threshold experiment plots hit rate against threshold on a random corpus
subsample. Corpus documents are semantically diverse by design — pairwise
similarity is low across the board. Real user paraphrase queries score
approximately 0.70-0.80. The value 0.75 captures genuine paraphrases
while excluding semantically distinct queries.

## 8. API Design

### FastAPI

Native async, automatic Swagger UI at `/docs`, Pydantic validation.

### Lifespan pattern

All heavy objects loaded once at startup — not per request.
Loading order: model → ChromaDB → memberships → cluster centers → PCA model.

### Swagger UI as interface

FastAPI generates an interactive UI at `http://localhost:8000/docs`.
All endpoints testable directly from the browser — no separate frontend required.
